# Dual Sizing Methodology — 이중 산출 방법론

> SCOPER (Phase 1) 가 방법론을 잠글 때, ANALYST (Phase 2) 가 시장 규모를 산정할 때, INTEGRATOR (Phase 4) 가 정합성을 검증할 때 참조.
> v1.3-v1.4 KR SaaS 리서치에서 확립·검증된 프레임워크.

---

## 1. 원칙: 이중 산출 (Dual Sizing)

**같은 시장 규모를 두 가지 독립적 방법으로 산정하고, 상호 검증합니다.**

| 방법 | 산정식 | 답하는 질문 |
|---|---|---|
| **A. Bottom-up** | 사업체 수 × SaaS 채택률 × 평균 연 SaaS 지출 | "사업체가 얼마나 **쓰는가**?" (지출 측) |
| **B. Solution 합** | Σ(각 솔루션 카테고리 매출) | "SaaS 회사가 얼마 **버는가**?" (매출 측) |

---

## 2. 왜 두 방법 모두 필요한가 (5가지 사유)

1. **본질적으로 다른 단면** — 지출 vs 매출, 어느 한쪽도 "정답" 아님
2. **데이터 출처 독립** — 정확히 같은 결과가 나오면 오히려 의심 (one-source bias)
3. **CEO 의사결정 다른 활용도**:
   - TAM 평가 → 방법 A (모집단 × 잠재 ARPU)
   - 카테고리별 매출 집중도 → 방법 B
   - M&A 타깃 점유율 → 방법 B
   - 채택률 변화 시 시장 규모 변동 시나리오 → 방법 A
4. **위키 검증 규칙 핵심 도구** — A→B→C 정합성 검증의 실제 적용
5. **차이 자체가 정보** — 차이 크면 caveat 영역 명시

---

## 3. 정합성 검증 기준

### 3.1 기본 기준: ±5%

| 차이 범위 | 판정 | 조치 |
|---|---|---|
| **±5% 이내** | ✅ 통과 | 두 출처 모두 신뢰 |
| **±5% ~ ±15%** | ⚠️ caveat | 차이 원인 명시 + 버티컬 자체 오차 범위 내 인지 확인 |
| **±15% 초과** | ❌ 정밀 조사 필요 | CHECKER-A 재실행 + 원인 식별 + 보고서 caveat 또는 수치 조정 |

### 3.2 차이 발생 일반 원인 (4가지)

1. **데이터 소스 독립성** — 다른 재무제표 항목 / 다른 회사 출처
2. **GMV vs 매출 정의 차이** — 양면 시장 (배달·OTA) 거래 수수료율 적용 차이
3. **통합 SaaS Type A 매출 분배** — 다버티컬 회사의 vertical 간 귀속 비율 추정
4. **Tier E 오차 누적** — 두 방법 모두 일부 추정 사용, 각자 ±25% 범위

### 3.3 검증 절차 (INTEGRATOR 의무)

```
1. 방법 A 합산 = Σ(버티컬별 Bottom-up 중앙값)
2. 방법 B 합산 = Σ(버티컬별 Solution Rows 합 중앙값)
3. 차이 (%) = (B - A) / A × 100
4. 버티컬별 개별 차이도 산정
5. ±5% 초과 시 원인 명시
```

## 4. 보고서 표기 규칙

### 4.1 § 3 시장 규모 표에 이중 산출 비교 footnote (의무)

각 버티컬 시장 규모 표 아래에:

```markdown
> **🆕 이중 산출 비교**: 방법 A (Bottom-up) USD XXM vs 방법 B (Solution 합 중앙값) USD YYM, 차이 **+Z.Z%** ✅/⚠️ 위키 ±5% [통과/caveat].
```

### 4.2 부록에 정합성 상세 매트릭스 (의무)

```markdown
| Vertical | A (USD M) | B (USD M) | 차이 (%) | ±5% 기준 | 비고 |
|---|---|---|---|---|---|
```
