# Classification Framework — 이중 분류 체계 가이드

> SCOPER (Phase 1) 가 분류 체계를 잠글 때, ANALYST (Phase 2) 가 표를 작성할 때 참조.
> v1.3-v1.4 KR SaaS 6 버티컬 리서치에서 도출·검증된 프레임워크.

---

## 1. 원칙: 이중 분류 (Dual Classification)

모든 시장 리서치에서 **두 가지 분류 축** 을 동시에 유지합니다:

| 축 | 역할 | 예시 |
|---|---|---|
| **1차 분류 (Primary)** | 시장 규모·점유율·매출 산출의 기본 단위 | Solution Rows (5-8 카테고리) |
| **보조 분류 (Secondary)** | MECE 검증·횡단 비교·위키 호환성 확보 | Function 축 (F1-F6) |

---

## 2. 1차 분류 채택 기준 (우선순위 순)

### 기준 1: 데이터 가용성 (최우선)

| 평가 | 정의 | 채택 기준 |
|---|---|---|
| ✅ High | Tier A·B 출처 (정부 통계·DART·회사 IR) 에서 **직접** 확인 가능 | 1차 분류 후보 |
| ⚠️ Medium | Tier B·C (IR + 매체) cross-reference 추정 가능 | 보조 분류 |
| ❌ Low | Tier D·E (자체 추정 多) ±25% 이상 오차 | 채택 불가 — 부록 보존 |

**판정법**: "이 분류 단위로 시장 규모를 산정할 때 사용하는 출처의 Tier 분포가 Tier A·B 70%+ 인가?"

### 기준 2: 청중 적합성

- CEO/이사회: "이 시장에 진출할 가치가 있는가?" → **상품 카테고리 단위** 유용
- CTO/R&D: "기능 갭은 어디인가?" → **Function 축** 유용
- 기본값: CEO/이사회 = 1차 분류는 상품 카테고리 우선

### 기준 3: 스코프 중복 회피

- 기존 보고서 스코프와 다른 분류가 완전 겹치면 빈 컬럼 양산 → 미채택
- 예: Front-of-house vs Back-of-house — 본 보고서가 이미 Front 만 다루면 분류 추가 무의미

### 기준 4: 위키/팀 표준 호환

- Workers-Hub 위키 backbone 정의가 있으면 **보조 분류로 반드시 포함** (호환성)
- 위키 정의와 충돌하는 분류는 caveat 명시

---

## 3. 미채택 분류 처리 (부록 보존 의무)

검토 후 채택하지 않은 분류 체계는 **부록에 반드시 보존**:

```markdown
## 부록 X. 미채택 분류 체계

### X.1 [분류명]

#### 정의
[표: 구분 | 정의 | 예시]

#### 미채택 사유 (3가지)
1. [사유 1 — 데이터 가용성/청중/중복/호환 중 하나]
2. [사유 2]
3. [사유 3]

#### 가능한 활용 시나리오
[향후 어떤 별도 보고서에서 활용 가능한지]
```

### KR SaaS 에서 미채택된 분류 3개 (참고)

| 분류 | 데이터 가용성 | 청중 적합성 | 미채택 사유 핵심 |
|---|---|---|---|
| Front vs Back-of-house | ❌ Low | ⚠️ Medium | 스코프 중복 (Front 만 다룸) |
| B2B vs B2C | ❌ Low | ⚠️ Medium | 양면 시장 — 같은 회사가 양쪽 |
| Stack-based (FE/BE/Infra) | ❌ Low | ❌ Low | Vertical SaaS 와 Layer 직교 |

---

## 4. 보조 분류의 활용 가치 (4가지)

| 가치 | 설명 |
|---|---|
| **MECE 검증** | 보조 축 합 = 시장 전체? → 1차 축과 ±5% 비교 (정합성 확인) |
| **위키 호환** | 팀 위키 backbone 정의와의 매핑 유지 |
| **횡단 비교** | 여러 버티컬을 동일 보조 축으로 비교 (예: "F2 가 강한 버티컬은?") |
| **기능 갭 분석** | Y/N 매트릭스 → 회사별 미진출 Function 식별 |

---

## 5. 데이터 가용성 평가 매트릭스 (SCOPER 가 작성)

SCOPER 가 Phase 1 에서 **각 버티컬 × 분류 축** 별 가용성을 평가한 표를 작성해야 합니다:

```markdown
| 버티컬 | 1차 분류 가용성 | 보조 분류 가용성 | 채택 |
|---|---|---|---|
| [버티컬 1] | ✅/⚠️/❌ (사유) | ✅/⚠️/❌ (사유) | [1차 + 보조 / 양쪽 동등 / 1차만] |
| [버티컬 2] | ... | ... | ... |
```

---

## 6. 적용 체크리스트

- [ ] SCOPER: 분류 축 2개 이상 후보 도출
- [ ] SCOPER: 데이터 가용성 평가 매트릭스 작성
- [ ] SCOPER: 1차·보조 확정 + 미채택 분류 보존 계획
- [ ] ANALYST: 1차 분류 기준 표 작성 + 보조 분류 매트릭스 병기
- [ ] INTEGRATOR: 1차·보조 합 정합성 ±5% 검증
- [ ] PACKAGER: 부록에 미채택 분류 + 채택 사유 종합 매트릭스 포함
